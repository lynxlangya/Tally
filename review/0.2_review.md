# 0.2 Review Report: Domain Constraints

## 1. 目标验证
- **关键枚举**: ✅ (`BillType`, `ExportFormat`, `ThemeMode` 已定义)
- **金额规范**: ✅ (`Money` 结构体，CNY，2位小数，禁止负数)
- **日期口径**: ✅ (`DayKeyFormatter` 统一产出 `yyyy-MM-dd`)

## 2. 检查项分析

### 2.1 代码逻辑正确性
- `Money` 结构体使用 `precondition(cents >= 0)` 强制保证非负数，逻辑正确。
- `MoneyFormatter` 强制使用 "zh_CN" Locale 和 CNY 货币代码，符合需求。
- `DayKeyFormatter` 使用 `en_US_POSIX` Locale 确保日期格式化不受用户设置影响，保证 Key 的稳定性。

### 2.2 可读性与风格
- 枚举定义清晰，实现了 `Identifiable` 和 `Codable`，利于后续使用。
- 工具类职责单一。

## 3. 风险评估
- **Crash Risk**: `Money(cents:)` 和 `MoneyFormatter` 中的 `precondition` 会在运行时触发 Crash（如果传入负数）。
  - **评估**: 这是设计上的强约束 ("必须写死")，在开发阶段暴露问题是合理的。但在 UI 层（如 `BillEditor`）必须确保输入校验逻辑完善，防止用户输入导致 Crash。

## 4. 结论
Domain 约束实现符合预期，格式化工具已就位。
